 \hypertarget{md_openbios_2README_autotoc_md12}{}\doxysection{\texorpdfstring{Purpose of this project}{Purpose of this project}}\label{md_openbios_2README_autotoc_md12}
There are several goals from the code in this directory.\hypertarget{md_openbios_2README_autotoc_md13}{}\doxysubsection{\texorpdfstring{Educational}{Educational}}\label{md_openbios_2README_autotoc_md13}
Writing an emulator requires extensive knowledge of the machine being emulated. The BIOS of the Play\+Station is part of it, and is obviously required to boot the machine and start games. This work is done so that both the author and the reader can perfect their understanding of its internals.

As a result, the code written here tries to be as close as possible to the original, bugs, quirks and all included. You may notice some anti-\/patterns while reading the code, like buffer overflows or arithmetic mistakes. They are intentional, and a direct reflection of what the retail BIOS does. Some of these bugs are weird enough that it\textquotesingle{}s difficult to infer the original buggy C code, while still understanding the general intent. Some of these bugs are severe enough that they have been fixed, but mentions are usually added to explain what changed and why.

Note that the generated binary will be nowhere close to the original rom this is inspired from. This is caused by several different factors\+: codestyle isn\textquotesingle{}t identical for a start, and some optimizations have been put in place still, such as the way syscalls are being issued. Linking order will also throw off some of the mappings. And finally, today\textquotesingle{}s compilers generate much smarter code than before. So the same C code won\textquotesingle{}t generate the same assembly code for two different compilers. There is also some dead or duplicated code in there that\textquotesingle{}d be useless or meaningless to reproduce. Finally, the shell itself is radically different.\hypertarget{md_openbios_2README_autotoc_md14}{}\doxysubsection{\texorpdfstring{Ease of distribution}{Ease of distribution}}\label{md_openbios_2README_autotoc_md14}
While dumping an original Play\+Station disc is something that is fairly easy to do for anyone using any sort of compact disc reader connected to a personal computer, dumping the bios of a retail machine, while not impossible, is a somewhat harder feat. As a result, people are often relying on bios dumps done by third party people, which ought to be considered copyright violation. Also, some emulators have a "{}\+HLE"{} bios, which creates a maintenance burden for each emulator, as well as being a complex mix of LLE and HLE code.\hypertarget{md_openbios_2README_autotoc_md15}{}\doxysubsection{\texorpdfstring{Testing}{Testing}}\label{md_openbios_2README_autotoc_md15}
Any recent project needs automated testing. Having source code available for all parts of the emulator will improve testing availability and test coverage, hopefully, for both the emulator, and the bios code.\hypertarget{md_openbios_2README_autotoc_md16}{}\doxysection{\texorpdfstring{Building}{Building}}\label{md_openbios_2README_autotoc_md16}
The currently supported build method for this project requires docker. You can use the {\ttfamily dockermake.\+sh} script at the root of the repository as a replacement for the {\ttfamily make} command, and use the {\ttfamily Makefile} present here. Under Windows, the {\ttfamily build.\+bat} can be used. If using \href{https://code.visualstudio.com/}{\texttt{ Visual Studio Code}}, one can use the task "{}make\+\_\+openbios"{} to compile\+: CTRL-\/P then {\ttfamily task make\+\_\+openbios} to compile.

It is also possible to build using any PS1 compatible compiler, but this isn\textquotesingle{}t the current method used in the CI, so it\textquotesingle{}s mostly untested. Check \doxysectlink{md_psyqo_2GETTING__STARTED}{PSYQo\textquotesingle{}s getting started}{0} document to install such toolchain in case docker isn\textquotesingle{}t an acceptable solution to build this project.

The result of the compilation should be a file called {\ttfamily openbios.\+elf} that contains all useful debugging symbols, and a file called {\ttfamily openbios.\+bin} which can be used in emulators or even burned to a chip and placed on a retail console.

The Makefile features several toggles to mutate the way the compilation works. Don\textquotesingle{}t forget to run {\ttfamily make clean} prior to try a different set of compilation options.


\begin{DoxyItemize}
\item {\ttfamily BUILD=Small\+Debug} will produce a working, yet somewhat debuggable version of the code.
\item {\ttfamily BOOT=cart} will build the kernel as an expansion port ROM, bootable from an Action Replay or similar cheat carts. This will disable support for booting from a cart in order to prevent bootlooping.
\item {\ttfamily BOARD=system573} will build a kernel that can run on the Konami System 573 arcade board. Note that this option is only useful alongside {\ttfamily EMBED\+\_\+\+PSEXE}, as it will force {\ttfamily BOOT=rom} and {\ttfamily BOOT\+\_\+\+MODE=psexe} due to the 573\textquotesingle{}s different CD-\/\+ROM hardware.
\item {\ttfamily EMBED\+\_\+\+PSEXE=binary.\+ps-\/exe} will embed the specified executable into the kernel and run it before the shell.
\item {\ttfamily BOOT\+\_\+\+MODE=fast} will skip running the shell and try booting from the CD-\/\+ROM after the embedded executable returns, or immediately on boot if {\ttfamily EMBED\+\_\+\+PSEXE} is not used. Note that the code will {\itshape not} wait for a disc to be inserted after the executable returns. The old {\ttfamily FASTBOOT=true} option is kept for compatibility and is equivalent to {\ttfamily BOOT\+\_\+\+MODE=fast}.
\item {\ttfamily BOOT\+\_\+\+MODE=psexe} will remove the shell and CD-\/\+ROM boot code altogether. This is meant to be used alongside {\ttfamily EMBED\+\_\+\+PSEXE} to build ROMs for arcade systems and such, which typically use non-\/standard storage devices and require a custom shell.
\item {\ttfamily SPLASH\+\_\+\+SCREEN=true} will make the kernel display color bars on screen during initialization. This is also done by some official non-\/retail BIOS variants and is useful when using the {\ttfamily EMBED\+\_\+\+PSEXE} option with a large (100+ KB) binary, as relocating it to RAM will take a couple of seconds.
\item {\ttfamily INSTALL\+\_\+\+TTY\+\_\+\+CONSOLE=true} will make Open\+BIOS install a DTL-\/\+H2000 host console driver in place of the default "{}dummy"{} TTY driver. Note that this is {\itshape not} the DUART driver found in a retail BIOS.
\end{DoxyItemize}\hypertarget{md_openbios_2README_autotoc_md17}{}\doxysection{\texorpdfstring{Status}{Status}}\label{md_openbios_2README_autotoc_md17}
This subproject is fairly functional and usable. Open\+BIOS does almost all the same things as the retail BIOS does when booting, and implements most of its features. \href{https://docs.google.com/spreadsheets/d/1UNGs7uYb8viAbm7YJaf1CR4dkgX7ZzntUdcowGsjcVc/edit?usp=sharing}{\texttt{ Many games}} are booting and working properly with this code. It can be used in emulators or on the real console, either while replacing the rom chip, or by using the "{}cart"{} build and programming the flash chip of a cheat cart with the result.

Please feel free to report bugs encountered while using this project.\hypertarget{md_openbios_2README_autotoc_md18}{}\doxysection{\texorpdfstring{Organization}{Organization}}\label{md_openbios_2README_autotoc_md18}
The BIOS is split in two major parts\+: the low level code for the bios itself, and the shell, which is the binary that\textquotesingle{}s being loaded into memory at boot time by the bios, to display the SONY sound and logo, and has a small utility menu for playing audio discs, or shuffling around memory cards.

While the first part is the main one that\textquotesingle{}s being targeted here, the second one hasn\textquotesingle{}t been reversed, and fully replaced with a completely different project. This may change in the future, but this isn\textquotesingle{}t currently the focus of this project.

The original code was most likely chunked into several sub-\/projects, that were all linked together like a giant patchwork. This approach is less readable, and for this reason, we\textquotesingle{}re not going to do this. However this will result in the ROM/\+RAM split to be less obvious, and slower at times than the original. Tuning of the hot functions is eventually required.

The startup point for the ROM version is the function {\ttfamily \+\_\+reset} located in \href{boot/psx.s}{\texttt{ {\ttfamily boot/psx.\+s}}} (or \href{boot/system573.s}{\texttt{ {\ttfamily boot/system573.\+s}}} depending on which board the kernel is compiled for). The rom cart version is the function {\ttfamily \+\_\+cart\+Boot} in the same file.\hypertarget{md_openbios_2README_autotoc_md19}{}\doxysection{\texorpdfstring{Direction}{Direction}}\label{md_openbios_2README_autotoc_md19}
The primary repository for this project is a subdirectory of PCSX-\/\+Redux at the moment, because building, testing and integration as a single bloc of code is much more practical than separate repositories. If the need arise however, it should be possible to move it as a separate repository elsewhere. There is currently a \href{https://github.com/pcsx-redux/nugget/tree/main/openbios}{\texttt{ buildable read-\/only mirror available}}.\hypertarget{md_openbios_2README_autotoc_md20}{}\doxysection{\texorpdfstring{Technicalities}{Technicalities}}\label{md_openbios_2README_autotoc_md20}
The code has been rewritten based off the reverse engineering of a dump of the BIOS of an american SCPH-\/7001 machine (MD5 {\ttfamily 1e68c231d0896b7eadcad1d7d8e76129}). The System 573 specific parts are based on the kernel of the Konami 700B01 ROM (MD5 {\ttfamily d6960997e499f39047b64fa73e6ff382}).

The ghidra database for it is currently being hosted on a server, alongside a few other pieces of software being reversed. Contact one of the authors if you want access.\hypertarget{md_openbios_2README_autotoc_md21}{}\doxysection{\texorpdfstring{Commentary}{Commentary}}\label{md_openbios_2README_autotoc_md21}
The retail Play\+Station BIOS code is a constellation of bugs and bad design. It is very obvious the code has been written hastily, likely by different teams having little to no communication with each other, under heavy time pressure. The fact that the retail console boots at all is nothing short of a miracle. Half of the provided libc in the A0 table is buggy. The BIOS code is barely able to initialize the CD-\/\+Rom, and read the game\textquotesingle{}s binary off of it to boot it; anything beyond that will be crippled with bugs. And this only is viable if you respect a very strict method to create your CD-\/\+Rom. The memory card and gamepad code is a steaming-\/hot heap of human excrement. The provided GPU stubs are inefficient at best. The only sane thing that any software running on the Play\+Station ought to do is to immediately disable interrupts, and then trash the whole memory to install its own code. Anything from the retail code is virtually useless, and shouldn\textquotesingle{}t be relied upon. That being said, doing so will prevent tools from functioning properly, like cheat software, unirom, or even emulators that are sniffing the kernel to do debugging and reporting.\hypertarget{md_openbios_2README_autotoc_md22}{}\doxysection{\texorpdfstring{Legality}{Legality}}\label{md_openbios_2README_autotoc_md22}

The same exact methodology was employed by Connectix for their PS1 bios. Accolade}} seems to indicate that this project here follows and is impacted by the same doctrine.
