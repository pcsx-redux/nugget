These two methods are called during the execution of the application. The first one is called before the application starts, when the hardware hasn\textquotesingle{}t been initialized yet, and the second one is called every time the scene stack is empty.\hypertarget{md_psyqo_2CONCEPTS_autotoc_md33}{}\doxysection{\texorpdfstring{The $<$tt$>$\+Scene$<$/tt$>$ class}{The <tt>Scene</tt> class}}\label{md_psyqo_2CONCEPTS_autotoc_md33}
The PSYQo library uses the concept of a {\ttfamily Scene} to represent the current state of the application. A minimum of one scene is needed to get the application to function properly.

The idea behind the scenes system is that state transitions is usually the hardest part of applications. By using the {\ttfamily Scene} class, we can easily create a scene stack, which will allow us to easily switch between different states of the application. Now, this doesn\textquotesingle{}t mean the scene stack is mandatory, but it is a good idea to use it. An application with only a single root scene is an acceptable outcome.

Both the {\ttfamily Scene} class and the {\ttfamily Application} class have the methods {\ttfamily push\+Scene} and {\ttfamily pop\+Scene} to push and pop scenes. The {\ttfamily push\+Scene} method will push its scene argument to the scene stack. The {\ttfamily pop\+Scene} method will remove the top scene from the stack, returning its pointer.

Now, this still won\textquotesingle{}t do much, but we\textquotesingle{}re making progress. The next step is to add some code to the {\ttfamily Scene} class, which will have to interact with the {\ttfamily GPU} class.\hypertarget{md_psyqo_2CONCEPTS_autotoc_md34}{}\doxysection{\texorpdfstring{The $<$tt$>$\+GPU$<$/tt$>$ class}{The <tt>GPU</tt> class}}\label{md_psyqo_2CONCEPTS_autotoc_md34}
Our third and last unavoidable class is the {\ttfamily GPU} class. The {\ttfamily GPU} class is basically the heartbeat of the application. There is no need to instantiate it as the {\ttfamily Application} class will do it for us. This means it\textquotesingle{}s possible to retrieve the {\ttfamily GPU} instance from the {\ttfamily Application} class using the method {\ttfamily gpu()}.

The class is responsible for drawing primitives on screen, and keeping the pace of the application. This means it also holds the timers for the application. It needs to be initialized from the {\ttfamily prepare} method of the {\ttfamily Application} class. This is done using a simple configuration system to create the rendering context.

When the application is running, the top scene on the stack will have its {\ttfamily frame} method called to draw the scene. This is the last piece of our puzzle in order to create a minimal working application. We can even easily create a simple animation using only what we have learned so far.

\hypertarget{md_psyqo_2CONCEPTS_autotoc_md35}{}\doxysection{\texorpdfstring{The scene stack}{The scene stack}}\label{md_psyqo_2CONCEPTS_autotoc_md35}
When creating an application with multiple scenes, only the top scene on the stack will be active and drawn. When pushing and popping scenes, they will get notified of the change. When a scene becomes active, its {\ttfamily start} method will be called. When a scene becomes inactive, its {\ttfamily teardown} method will be called. Both methods have a {\ttfamily reason} argument, which can be used to determine the reason for the change.

When a scene is started, it can be because it just got pushed to the stack (which is the {\ttfamily Create} reason), or because it became the top scene on the stack due to a call to {\ttfamily pop\+Scene} (which is the {\ttfamily Resume} reason).

When a scene is teardown, it can be because it just got popped from the stack (which is the {\ttfamily Destroy} reason), or because a new scene arrived on top of the stack due to a call to {\ttfamily push\+Scene} (which is the {\ttfamily Pause} reason).

By encapsulating scene states that way, we can simplify the general approach of state transitions within the whole application.\hypertarget{md_psyqo_2CONCEPTS_autotoc_md36}{}\doxysection{\texorpdfstring{Primitives}{Primitives}}\label{md_psyqo_2CONCEPTS_autotoc_md36}

This way, the fragment will be initialized and filled with the primitives only once, and then reused for every frame, with only the necessary changes being applied.

Note that the fragment is not copied when it is sent to the GPU, so it is not safe to mutate the fragment within event callbacks, as they can be dispatched during {\ttfamily send\+Fragment}. This could potentially create visual glitches as the fragment could be modified while it\textquotesingle{}s being sent to the GPU.\hypertarget{md_psyqo_2CONCEPTS_autotoc_md39}{}\doxysection{\texorpdfstring{Fragment chaining}{Fragment chaining}}\label{md_psyqo_2CONCEPTS_autotoc_md39}
Fragments can be chained together to create a sequence of primitives to draw. The advantage here is that the code no longer has to wait on {\ttfamily send\+Fragment} to finish, before moving on to the next phase of computation. Instead, the chained fragments will be sent in the background to the GPU in the order they were chained, and the code can continue executing. The actual sending of the fragments will start when the {\ttfamily frame} method returns, and the GPU is ready to accept new fragments. The chain will continue to be sent out to the GPU when the next call to {\ttfamily frame} is made, so double buffering of the fragments becomes necessary. In order to help with this, the {\ttfamily GPU} class has a {\ttfamily get\+Parity} method, which will return the current parity of the GPU, which can be used to determine which buffer to write to.

An important limitation of the chaining mechanism is that the maximum size of a fragment is 255 words. This is a limitation of the PS1 hardware, and not of the PSYQo library. The library will assert if the fragment size exceeds this limit when being chained. Note that it is still possible to send larger fragments using {\ttfamily send\+Fragment}, but care must be taken to do so only when the previous chain has been fully sent. This also means that once the user decides to use chaining, they must use it for all fragments, and not mix and match with {\ttfamily send\+Fragment}. This means using a different general paradigm for the drawing code.

Other than the size limitation, one simply need to call the {\ttfamily chain} method of the {\ttfamily GPU} class, and pass the fragment to chain.\hypertarget{md_psyqo_2CONCEPTS_autotoc_md40}{}\doxysection{\texorpdfstring{Ordering Tables}{Ordering Tables}}\label{md_psyqo_2CONCEPTS_autotoc_md40}
When doing 3D rendering, since the PS1 GPU doesn\textquotesingle{}t have a Z-\/buffer, the order in which primitives are drawn is important. The PSYQo library provides a way to sort primitives based on their Z position, using the {\ttfamily Ordering\+Table} class. The class is a template taking a number of "{}buckets"{} as a parameter. When inserting a primitive into the ordering table, one needs to provide a z value, which represents the depth of the primitive. This z value will be used to select which bucket the primitive will be inserted into. No ordering is done within the buckets, so primitives with the same z value will be drawn in the order they were inserted, but the buckets themselves will be drawn in order of decreasing z value, farthest bucket first, closest bucket last. When inserting a primitive inside an {\ttfamily Ordering\+Table\texorpdfstring{$<$}{<}N\texorpdfstring{$>$}{>}}, the z value should be in the range of 0 to N-\/1, with 0 being the closest to the camera, and N-\/1 being the farthest. If the z value is outside of this range, the primitive will be inserted into the closest bucket, with the z value clamped to the range.

The {\ttfamily Ordering\+Table} class can only be sent to the GPU using chaining. While not being a Fragment itself, it can be chained with other fragments. Multiple ordering tables can also be chained within the same scene, and they will be sent to the GPU in the order they were chained. This also means it is essential to double buffer the ordering tables, as they will be sent to the GPU in the background, and the code will continue executing.

Once the ordering table has been sent to the GPU, it will be cleared, and can be reused.\hypertarget{md_psyqo_2CONCEPTS_autotoc_md41}{}\doxysection{\texorpdfstring{Memory management}{Memory management}}\label{md_psyqo_2CONCEPTS_autotoc_md41}
Note that none of the current \mbox{[}examples\mbox{]}(examples) are currently using any memory allocation, unless explicitly showcasing that memory allocation works. The core library itself may allocate memory, when it needs to overspill some heavy usage cases, but it should not be the general case.

Memory allocation in general with such a small amount of available memory is not necessarily a good idea, so it is generally recommended to avoid it. But all of the normal C++ memory allocation primitives should be working. Note that no standard libc is provided, so function calls like {\ttfamily malloc} and {\ttfamily free} are not directly available. The {\ttfamily psyqo\+\_\+malloc} and {\ttfamily psyqo\+\_\+free} functions are provided instead, and are the foundation of the {\ttfamily operator new} and {\ttfamily operator delete} functions.\hypertarget{md_psyqo_2CONCEPTS_autotoc_md42}{}\doxysection{\texorpdfstring{Concurrency}{Concurrency}}\label{md_psyqo_2CONCEPTS_autotoc_md42}
The major design principle of the PSYQo library is asynchronous callbacks. Most of the library is designed to be used in an asynchronous manner, and thus many of the functions are non-\/blocking. The only parts of the API which may be synchronous are inside the GPU subsystem, as it is the one designed to set the tempo of the application. Callbacks may be dispatched from any blocking GPU operation, and between frames. Some operations with asynchronous callbacks may complete successfully during the scheduling of the operation, and thus the callback may be dispatched immediately, from the same callstack as the scheduling method. Users of the library should be aware of this, and should not rely on callbacks always being dispatched at a later time.

In addition to asynchronous callbacks, there are facilities designed to help writing more readable asynchronous code.\hypertarget{md_psyqo_2CONCEPTS_autotoc_md43}{}\doxysubsection{\texorpdfstring{Coroutines}{Coroutines}}\label{md_psyqo_2CONCEPTS_autotoc_md43}
C++20 introduced support for coroutines. Coroutines are a way to write asynchronous code in a synchronous manner. They are a very powerful tool, and can be used to write very readable asynchronous code. \href{https://pcsx-redux.github.io/nugget/d6/d6f/structpsyqo_1_1Coroutine.html}{\texttt{ PSYQo has support for these coroutines}}, and there is \href{examples/coroutine-demo}{\texttt{ an example}} showcasing their usage. One important caveat is that the language will silently and implicitly use memory allocation to store the coroutine state, which may be a problem for some users. This is a limitation of the C++ language, and not of PSYQo.

In order to properly use coroutines, the function will need to have access to its coroutine object, so passing a pointer as a parameter that can give access to the coroutine object is necessary. The coroutine object can then be used to suspend the coroutine, and to resume it at a later time, usually during scheduled callbacks. For example\+:

The above coroutine will perform an asynchronous operation, and then return the result of that operation. The coroutine will be suspended until the asynchronous operation completes, and then resumed by its callback. The result will be held inside of the coroutine object for the caller to inspect. The {\ttfamily co\+\_\+await} and {\ttfamily co\+\_\+return} keywords are specific to coroutines, and are part of the C++ language as defined by the C++20 revision. The {\ttfamily co\+\_\+await} keyword will suspend the coroutine, and the {\ttfamily co\+\_\+return} keyword will terminate the coroutine and store the result inside of the coroutine object.

The coroutine object will be in a "{}done"{} state after the coroutine has terminated, and can be reused by assigning a new coroutine to it. The {\ttfamily value} method will return the result of the coroutine, and can only be called after the coroutine has terminated. The {\ttfamily resume} method will start the coroutine, or resume it if it has been suspended. The {\ttfamily done} method will return {\ttfamily true} if the coroutine has terminated, and {\ttfamily false} otherwise. The coroutine starts in a suspended state, and must be resumed to begin operations. Resuming a coroutine that has already terminated will have no effect. Resuming a coroutine that is already running will likely end up in corruption of the coroutine state, and will likely result in a crash.\hypertarget{md_psyqo_2CONCEPTS_autotoc_md44}{}\doxysubsection{\texorpdfstring{Task queue}{Task queue}}\label{md_psyqo_2CONCEPTS_autotoc_md44}
If the above coroutine system is too expensive, there is a less capable but more efficient way to still have a readable asynchronous code. The \href{https://pcsx-redux.github.io/nugget/d4/db5/classpsyqo_1_1TaskQueue.html}{\texttt{ Task\+Queue}} class allows for scheduling in advance a series of synchronous or asynchronous tasks that will be executed sequentially, loosely inspired by the Java\+Script Promise system. There is a \href{examples/task-demo}{\texttt{ an example}} showcasing its usage.

This system does not allow for loops or conditions, but will consume less resources overall than coroutines. It can be beneficial to use this system if the code is not expected to be very complex.\hypertarget{md_psyqo_2CONCEPTS_autotoc_md45}{}\doxysection{\texorpdfstring{Further reading}{Further reading}}\label{md_psyqo_2CONCEPTS_autotoc_md45}
The \href{examples/tetris}{\texttt{ Tetris}} example has a thorough usage of the library so far, and is a great example of how to use it. The library itself is also thoroughly documented, and the \href{https://pcsx-redux.github.io/nugget/d6/d4a/namespacepsyqo.html}{\texttt{ nugget website}} has a render of the doxygen documentation.
